> 研究日期：2026-06-25

## 一、Google — g3doc + Moma

Google 内部的知识管理体系是业界架构设计的标杆，核心理念是**文档与代码同位（co-location）**。

### 内部系统全景

```
Google 内部知识体系
├── g3doc（代码文档系统）★ 最有参考价值
│   ├── 文档与代码存放在同一仓库目录下
│   ├── 自动生成 API 文档（从代码注释）
│   └── 代码变更触发文档更新 Review 提醒
├── Moma（内部搜索引擎）
│   ├── 搜索范围：代码、文档、邮件、日历、人员
│   ├── 知识卡片（人员名片、项目概要、系统信息）
│   └── 2023 年后 Bard/Gemini 接入，支持自然语言问答
└── Sites（内部 Wiki）
    └── 基于 Google Sites，各团队自建知识空间
```

### g3doc 设计精髓（★ 值得借鉴）

```
g3doc 文件结构示例
my-service/
├── src/
│   └── main.go
├── BUILD              ← 声明文档路径，CI 检查覆盖率
└── g3doc/
    ├── README.md      ← 服务说明
    ├── design.md      ← 设计文档
    └── oncall.md      ← On-call 手册
```

**核心规则：**
- 文档和代码**强绑定**：代码 PR 合并时，CI 检查是否同步更新了 g3doc
- 文档覆盖率是 Code Review 的指标之一
- 这一设计保证文档不会滞后于代码，解决了大多数团队的文档腐化问题

**影响力：** 这一模式影响了 Bazel 生态、大量 Google 开源项目，以及 Spotify 的 Backstage TechDocs。

## 二、Meta — Workplace + Bento

### 核心系统

```
Meta 知识管理体系
├── Workplace（企业版 Facebook）
│   ├── 内部 IM + 动态流
│   ├── 知识文章（Knowledge Library）
```

- 内部 AI 助手可以回答"这个函数是谁写的/为什么这么设计"

## 三、Netflix — 文档文化 + Backstage

Netflix 的知识管理以**工程文化**著称：每个系统必须有完整的 Runbook，文档是工程师 On-call 的基础保障。

### 核心体系

```
Netflix 知识管理体系
├── Confluence（主 Wiki，团队文档）
├── Backstage（服务目录，Spotify 开源）★
│   ├── 每个微服务有标准化"说明书"
│   │   ├── API 文档
│   │   ├── 依赖关系图
│   │   ├── On-call 负责人
│   │   └── 健康状态 Dashboard
│   └── TechDocs（Markdown → 自动发布为文档站）
└── 内部 RAG 问答（2024 年上线）
    └── 基于 OpenAI API + 内部 Confluence/Runbook 的 RAG
```

### Backstage 的服务知识库理念

Backstage（已开源，GitHub 55k+ Star）定义了"服务目录（Service Catalog）"的标准：

```yaml
# catalog-info.yaml（每个服务根目录必须有）
apiVersion: backstage.io/v1alpha1
kind: Component
metadata:
  name: my-service
  description: "外卖订单处理服务"
  annotations:
    github.com/project-slug: org/my-service
spec:
  type: service
  lifecycle: production
  owner: team-order
  system: order-platform
```

这个 YAML 是服务的"知识名片"，Backstage 聚合所有服务的 catalog-info.yaml，自动生成可搜索的服务知识库。

## 四、Atlassian — Confluence 的 AI 进化

Confluence 是企业知识库的**行业默认标准**，2024年的 AI 升级标志着传统 Wiki 向 AI-Native 的转型。

### Confluence AI（Atlassian Intelligence，2024）

```
Confluence AI 能力矩阵
├── 内容理解
│   ├── 页面摘要（一键生成，支持长文档）
│   ├── Action Items 提取（自动识别文档中的待办事项）
│   └── 知识库 Q&A（基于 Space 内容的 RAG 问答）
├── 写作辅助
│   ├── 扩写、改写、翻译
│   ├── 从大纲生成完整页面
│   └── 变更日志自动生成（结合 Jira Issue）
├── 语义搜索
│   ├── 向量化所有页面内容
│   ├── 自然语言查询（不需要关键词）
│   └── 答案溯源到页面 + 段落链接
└── 与 Jira 打通
    ├── Issue 关闭 → 自动提示更新关联 Wiki
    └── Sprint 结束 → 自动生成回顾知识文档
```

### 关键设计：Jira ↔ Confluence 知识闭环

```
需求（Jira） → 设计文档（Confluence） → 代码实现 → Issue 关闭
     ↑                                                    │
     └──────── AI 自动生成复盘文档 ←────────────────────┘
```

这是 Atlassian 最大的差异化：**任务管理和知识管理的双向绑定**。

## 五、Notion — 从文档工具到 AI 知识平台

### Notion AI Q&A（2023-2024 核心功能）

```
Notion AI 知识库能力
├── Q&A 问答（★ 核心）
│   ├── 基于整个 Workspace 内容的 RAG 问答
│   ├── "Ask Notion AI" 自然语言入口
│   └── 引用来源精确到 Page + Block 级别
├── 文档操作
│   ├── 摘要、翻译、改写、续写
│   ├── 从空白生成文档（基于模板 + 上下文）
│   └── 数据库视图的 AI 填充
└── 知识连接器（2024 新增）
    ├── 接入 Slack、Google Drive、GitHub
    └── 跨平台知识统一问答（不只是 Notion 内容）
```

### 技术底层（已公开信息）

| 组件 | 选型 |
|------|------|
| Embedding 模型 | OpenAI `text-embedding-3-large` |
| 向量数据库 | **pgvector**（PostgreSQL 扩展） |
| 检索策略 | 语义检索 + BM25 混合检索 |
| 分块粒度 | **Block 级别**（非整页），保留块结构信息 |
| 生成模型 | GPT-4o（通过 OpenAI API） |

**pgvector 选型原因：** Notion 已深度使用 PostgreSQL，pgvector 作为扩展接入，避免引入新的基础设施，运维成本低。这是"够用即可，不过度复杂化"的典型工程决策。

### Block 级分块的优势

```
整页分块（传统方式）
  页面：《服务架构设计》（3000字）
  → 分块结果：[3000字整体]  ← 向量语义模糊

Block 级分块（Notion 方式）
  页面：《服务架构设计》
  ├─ Block: "## 存储层设计" + 后续段落  → 向量①
  ├─ Block: "## 网络层设计" + 后续段落  → 向量②
  └─ Block: "## 监控方案" + 后续段落    → 向量③
  
  查询"存储用什么数据库" → 精准命中向量①
```
