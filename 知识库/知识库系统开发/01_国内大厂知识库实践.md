# 国内大厂知识库实践

> 研究日期：2026-06-25

---

## 一、字节跳动 — 飞书知识库（Lark Wiki）

**产品定位：** 从字节内部实践孵化，对外商业化的企业知识管理平台

### 技术架构

```
飞书知识库架构
├── 存储层
│   ├── 文档树形结构（Space → Folder → Page）
│   ├── OT（Operational Transformation）实时协同引擎
│   └── 块级存储（Block-based，类 Notion）
├── 检索层
│   ├── Elasticsearch 全文检索
│   ├── 向量化索引（2023 年后接入）
│   └── 权限感知搜索（搜索结果按 ACL 过滤）
└── AI 层（2023-2024 年接入）
    ├── 飞书智能伙伴（My AI）
    ├── 基于企业知识库的 RAG 问答
    └── 文档摘要、改写、续写
```

### 核心设计亮点

- **三级空间隔离**：知识空间、团队空间、个人空间，权限精确到页面级别
- **OT 实时协同**：多人同时编辑不冲突，字节自研 OT 算法
- **生态深度绑定**：会议纪要自动生成 Wiki、IM 消息一键转文档

### AI 知识库能力（2024年）

- 基于企业私有知识库的 RAG 问答，答案**溯源到原始文档段落**
- 支持跨 Wiki、跨云文档的统一语义检索
- 知识库"问答助手"内嵌进搜索框，替代关键词检索

## 二、阿里巴巴 — 语雀 + 内部 ATA

**产品路线：** 内部 ATA（Alibaba Technology Articles）沉淀 → 2019 年语雀对外开放

### 内部 ATA 知识管理体系

```
ATA 架构
├── 内容组织
│   ├── 文章（技术博客形式）
│   ├── 知识库（结构化文档目录）
│   └── 话题（标签体系，横切关注点）
├── 社交机制
│   ├── 关注、点赞、收藏、评论
│   ├── 内部技术专栏（类 Medium）
│   └── 技术圈子（DDD 领域专属讨论）
└── 搜索与推荐
    ├── 全文检索（Elasticsearch）
    └── 个性化推荐（协同过滤，基于阅读行为）
```

### 语雀对外产品技术特色

- **结构化文档**：表格文档、画板、思维导图**原生支持**（非插件）
- **知识库版本管理**：文档 diff 对比、历史版本一键回滚
- **语雀 AI 助手（2023-2024）**：接入通义千问，支持知识库问答

### 语雀的核心产品理念

语雀是国内最早做"知识库"概念落地的产品。区别于 Confluence 的 Wiki 模式，语雀更强调**知识的结构化组织**而非纯文档堆积——每个知识库有明确的目录树，内容之间有引用关系，而非平铺的页面集合。

## 三、腾讯 — 企业微信知识库 + KM 内部系统

**技术路线：两条线并行**

```
腾讯知识管理双轨
├── 腾讯文档（对外 C 端/B 端）
│   ├── 在线 Office（Word/Excel/PPT 形态）
│   ├── 智能文档（问卷、收集表单）
│   └── AI 助手（混元大模型加持，2024）
└── 企业微信知识库（对 B 端）
    ├── 文档库（类 SharePoint 的文件管理）
    ├── 微盘（企业文件云存储）
    └── 企业知识库问答（2023 年上线，RAG）
```

### 内部 KM 系统

腾讯内部技术知识平台称为 **KM（Knowledge Management）**，是腾讯最早的技术知识沉淀平台：
- 基于 **Elasticsearch + MySQL** 混合存储
- 内容形态以**技术博文**为主，类似外部的掘金/博客园
- 2024 年接入混元大模型，支持"问 KM"自然语言检索

## 四、美团 — 技术知识体系

### 核心平台

| 平台 | 定位 | 技术栈 |
|------|------|--------|
| 内部 Wiki | 团队文档管理 | Confluence 二次定制 |
| tech.meituan.com | 对外技术沉淀 | 静态博客 |
| Databook | 数据治理知识库 | 自研（数据字典+血缘） |
| 智能客服知识库 | 客服 QA 问答 | 结构化 QA + 检索 |

### 2024 年 RAG 知识库实践（公开技术博客）

```
美团 RAG 知识库架构
├── 文档解析层
│   ├── PDF / Word / Markdown 多格式解析
│   ├── 表格、图片内容提取（OCR + 版式分析）
│   └── 分块策略优化（语义边界 > 固定大小）
├── 向量化层
│   ├── Embedding 模型（自研 + BGE 开源）
│   ├── 向量数据库：Milvus
│   └── 混合索引：稠密向量 + BM25 稀疏向量
├── 检索增强层
│   ├── Query 改写（HyDE 假设文档嵌入、多查询扩展）
│   ├── Rerank 精排（交叉编码器，BGE-Reranker）
│   └── 知识融合（多路检索结果去重合并）
└── 生成层
    ├── 美团自研大模型（推荐领域微调）
    └── 答案引用溯源（Citation 格式输出）
```

### 关键实践经验

1. **分块粒度**：推荐场景文档切 256-512 token，保留段落完整性优于固定切分
2. **Rerank 必要性**：向量召回 Top-50，Rerank 精排到 Top-5，效果提升显著
3. **文档质量治理**：引入文档新鲜度评分，过期文档降权

## 五、滴滴 — 知识图谱驱动

**特色方向：知识图谱（KG）优先，非纯向量方案**

### DIKW 体系

滴滴内部采用 **DIKW 体系**（Data → Information → Knowledge → Wisdom）构建知识管理：

```
滴滴知识体系
├── Data 层：原始运营数据、行程数据
├── Information 层：数据报表、监控指标
├── Knowledge 层
│   ├── 出行领域知识图谱（路网、POI、规则）
│   ├── 技术知识库（架构文档、规范）
│   └── 图数据库：Neo4j + 自研图存储
└── Wisdom 层（2024）
    └── LLM + 图谱混合问答
        ├── 自然语言 → Cypher 查询
        └── 图谱检索结果 + RAG 联合生成
```

### 图谱 + RAG 混合方案

- 实体关系类问题（"A 系统依赖哪些中间件"）→ 走图谱查询
- 开放性文档类问题（"如何处理 X 异常"）→ 走 RAG
- 复杂推理类问题 → 图谱提供结构 + RAG 提供细节，联合生成答案
