## 一、现代知识库系统标准架构（RAG 2.0）

```
┌─────────────────────────────────────────────────────────────────┐
│                   企业知识库 RAG 标准架构                        │
├─────────────────────────────────────────────────────────────────┤
│  文档摄入层（Ingestion Pipeline）                               │
│  ├─ 多格式解析：PDF / Word / Markdown / HTML / 代码             │
│  ├─ 表格 / 图片理解：OCR + 多模态模型（GPT-4V / Qwen-VL）      │
│  ├─ 分块策略：固定大小 / 语义分块 / 按文档结构分块              │
│  └─ 元数据提取：作者、时间、部门、标签、访问权限                │
├─────────────────────────────────────────────────────────────────┤
│  索引层（Dual Index）                                           │
│  ├─ 稠密向量：Embedding 模型 → 向量数据库                       │
│  ├─ 稀疏向量：BM25 / TF-IDF → 倒排索引                         │
│  └─ 图索引：知识图谱（实体-关系），适用于结构化知识             │
├─────────────────────────────────────────────────────────────────┤
│  检索层（Hybrid Retrieval）                                     │
│  ├─ Query 理解：意图识别、实体抽取、查询改写（HyDE / 多查询）   │
│  ├─ 混合检索：稠密 + 稀疏 + 图检索，RRF 融合排序               │
│  ├─ Rerank 精排：交叉编码器（BGE-Reranker / Cohere Rerank）     │
│  └─ 权限过滤：ACL 验证，过滤无权访问的文档块                   │
├─────────────────────────────────────────────────────────────────┤
│  生成层（Augmented Generation）                                 │
│  ├─ 上下文构建：检索片段 + 对话历史 + 系统提示词                │
│  ├─ LLM 推理：GPT-4o / Claude / 自研大模型                      │
│  ├─ 答案生成：带 Citation（来源引用）的结构化回答               │
│  └─ 幻觉检测：NLI 模型验证答案与来源的一致性                   │
└─────────────────────────────────────────────────────────────────┘
```

## 二、向量数据库选型对比

| 产品 | 公司背景 | 核心优势 | 适用规模 | 国内大厂采用 |
|------|----------|----------|----------|-------------|
| **Milvus** | Zilliz（国内） | 亿级向量、高并发、云原生 | 大规模 | 美团、百度、爱奇艺 |
| **Qdrant** | Qdrant 公司 | Rust 实现、延迟低、API 简洁 | 中小规模 | 创业公司居多 |
| **Weaviate** | SeMI Technologies | 知识图谱 + 向量混合原生支持 | 中规模 | 外资企业 |
| **Chroma** | Chroma（开源） | 本地开发友好，嵌入式 | 小规模/开发 | 原型验证 |
| **pgvector** | PostgreSQL 扩展 | 零新增基础设施，运维简单 | 小~中规模 | Notion、初创 |
| **Elasticsearch** | Elastic | 全文检索老牌，向量能力后加 | 全规模 | 几乎所有大厂 |
| **自研向量引擎** | 各大厂 | 超大规模定制化 | 超大规模 | 百度、阿里、字节 |

### 选型决策树

```
是否已有 PostgreSQL 且规模 < 1亿向量？
  ├─ 是 → pgvector（零运维成本）
  └─ 否 → 是否需要知识图谱混合？
           ├─ 是 → Weaviate
           └─ 否 → 规模 > 5亿？
                   ├─ 是 → Milvus（或自研）
                   └─ 否 → Qdrant（性能好，运维简单）
```

## 四、RAG 开发框架对比

```
RAG 框架生态（2024-2025 主流选型）

├── LangChain（最流行，Python）
│   优势：生态最大，工具链最全
│   劣势：太重，抽象层多，生产稳定性差，升级频繁破坏性大
│   适用：原型开发、快速验证

├── LlamaIndex（文档处理更专注）
│   优势：Document Store 抽象完善，分块/索引策略丰富
│   劣势：生产案例比 LangChain 少
│   适用：以文档问答为核心的 RAG 系统

├── Dify（国内最流行的 RAG 产品化平台）★
│   优势：可视化编排，开源+私有部署，大量企业实际在用
│   劣势：定制化能力受限于平台
│   适用：快速上线企业知识库，不想从零写代码

├── RAGFlow（国内 InfiniFlow 出品）★
│   优势：文档解析最强（表格、版式保留），视觉理解
│   劣势：较新，社区相对小
│   适用：文档质量要求高的场景（金融、法律、技术文档）

├── FastGPT（国内开源）
│   优势：部署最简单，面向知识库问答优化
│   劣势：功能相对单一
│   适用：个人/小团队快速搭建知识库问答

└── 自研框架
    适用：大厂，对流程控制要求高，有专门 AI 工程团队
```

## 五、企业知识库最难点：权限感知 RAG

权限隔离是**企业级知识库落地最大的工程挑战**。

### 问题描述

```
风险场景：
  普通员工 → 问 AI "公司今年的薪酬调整方案"
  AI 从训练/索引的高密级 HR 文档中学到了答案
  → 泄露敏感信息

根源：向量检索本身不理解权限，只看语义相似度
```

### 三种解决方案

**方案 A：检索后过滤（Post-retrieval ACL Filter）**

```
检索 Top-K 候选 → 对每个候选查 ACL → 过滤无权限文档 → 剩余喂给 LLM

优点：实现简单，改动小
缺点：Top-K 可能被全部过滤，召回率下降
     过滤逻辑在检索后，有一定信息泄露风险（向量相似度本身暴露了文档存在）
适用：权限粒度粗（部门级别），文档敏感度差异不大的场景
```

**方案 B：索引隔离（Per-role Index）**

```
为每个角色/用户组维护独立的向量索引
用户查询 → 只检索本人有权限的索引

优点：安全彻底，无泄露风险
缺点：存储成本 = 原始存储 × 角色数
     文档更新需同步更新所有相关索引，维护成本极高
适用：极高安全要求（军事、金融核心系统）
```

**方案 C：元数据过滤（Metadata Filter）★ 主流方案**

```
文档入库时，将权限信息作为元数据存储：
{
  "vector": [...],
  "content": "...",
  "metadata": {
    "acl": ["team:hr", "role:director"],  ← 权限标签
    "classification": "confidential"
  }
}

检索时附加过滤条件：
query_vector + filter(acl contains user.roles)

优点：工程可行，性能影响小（大多数向量数据库原生支持元数据过滤）
     权限粒度灵活（文档级、段落级均可）
缺点：需要权限系统与知识库打通，文档权限变更时需更新元数据
适用：字节飞书、企业微信等企业产品的主流做法
```

## 六、文档分块策略详解

分块（Chunking）是 RAG 效果的关键，策略选择影响检索精度。

### 各策略对比

| 策略 | 原理 | 适用文档类型 | 优缺点 |
|------|------|-------------|--------|
| **固定大小** | 按 Token 数切分（如 512 Token，50 Token 重叠） | 所有类型 | 简单，但可能切断语义 |
| **按句子/段落** | 按自然语言边界（句号、段落符）切分 | 叙述型文档 | 语义完整，大小不均匀 |
| **按结构** | 按标题层级切分（H1/H2/H3 边界） | Markdown、技术文档 | 语义最完整，适合技术 Wiki |
| **语义分块** | 用 Embedding 检测语义突变点 | 长文档 | 效果最好，计算成本高 |
| **Block 级**（Notion） | 按文档 Block 结构切分 | 块编辑器（Notion/飞书） | 保留编辑结构，粒度细 |
| **父子分块** | 小块检索 + 大块生成（LlamaIndex 方案） | 技术文档 | 召回精准 + 上下文充分 |

### 父子分块（Parent-Child Chunking）详解 ★

这是目前实践效果最佳的策略，被 LlamaIndex 推广：

```
文档：《系统架构设计》（长文）
│
├─ 父块（512 Token，用于生成上下文）
│   └─ 子块1（128 Token，用于向量检索）
│   └─ 子块2（128 Token，用于向量检索）
│   └─ 子块3（128 Token，用于向量检索）
│
├─ 父块（512 Token）
│   └─ 子块4
│   └─ 子块5

查询时：
  用子块的向量做精确匹配（粒度细，命中率高）
  返回给 LLM 的是父块（上下文充分，不截断语义）
```

## 七、GraphRAG：知识图谱 + 向量混合检索

微软于 2024 年开源 GraphRAG，引发业界广泛跟进。

### 核心思路

```
纯向量 RAG 的局限：
  "A 服务和 B 服务是什么依赖关系？" → 向量检索找不到
  "哪些服务会影响结算模块？" → 需要图结构推理

GraphRAG 方案：
  第一步：对所有文档进行实体抽取、关系抽取 → 构建知识图谱
  第二步：对图进行社区检测 → 生成"社区摘要"（层级概要）
  第三步：查询时：
    本地查询 → 向量检索（原始文档片段）
    全局查询 → 图社区摘要（宏观理解）
    混合查询 → 两者结合
```

### 国内大厂的 GraphRAG 实践

| 公司 | 实践方向 |
|------|----------|
| 阿里 | 电商知识图谱（商品-品类-属性关系）+ RAG |
| 百度 | 百科知识图谱（文心一言的外挂知识） |
| 华为 | 技术文档知识图谱（API 依赖关系） |
| 滴滴 | 出行领域知识图谱（路网、POI、规则） |
