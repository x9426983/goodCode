
各阶段耗时拆解

真实请求处理完毕
      ↓
① Producer 异步线程池发送到 Mafka      → 几ms（异步，不阻塞主流程）
      ↓
② Mafka 网络传输 + Broker 写入          → 几ms ~ 几十ms
      ↓
③ Consumer 批量等待                     → 0ms ~ 2000ms ← 最大的不确定性
      ↓
④ ModelResetConsumer 处理 + 异步提交    → 几ms
      ↓
⑤ RankJobThreadPool 调用 rec-hub       → 取决于rec-hub响应时间


第③步是关键：批量消费配置

<entry key="mafka.batch.recv"             value="true"/>
<entry key="mafka.batch.recv.size"        value="200"/>   <!-- 凑满200条就消费 -->
<entry key="mafka.batch.recv.timeout.ms"  value="2000"/>  <!-- 最多等2秒 -->

消费者等待满足任意一个条件就触发消费：

条件A：积累了 200 条消息  → 流量大时可能几十ms就触发
条件B：等待超过 2000ms   → 流量小时最长等2秒

实际时延范围

┌─────────────────────────┬─────────────────┐
│          场景           │     总时延      │
├─────────────────────────┼─────────────────┤
│ 流量大（频繁凑满200条） │ 几十ms ~ 几百ms │
├─────────────────────────┼─────────────────┤
│ 流量小（靠超时触发）    │ 约 2000ms       │
└─────────────────────────┴─────────────────┘

---
但这个时延根本不重要

重放的目的是模型效果评估，不是实时响应用户：

原始请求  ──→  rec-hub（当前模型）──→ 结果返回给用户   ← 用户感知
↓ 复制
MQ（等待0~2s）
↓
rec-hub（新模型）──→ 结果丢弃           ← 用户无感知

新模型的排序结果只是被记录下来用于离线比对，晚1秒还是晚2秒对评估结果没有任何影响。批量消费本身是为了减少 MQ 消费的频繁 poll 开销，是合理的权衡。
