Mafka 消息重放机制全景分析

---
一、系统定位

目标：用真实线上流量对排序模型进行影子验证，在完全不影响用户体验的前提下，评估新模型效果。

两个服务的分工：

waimai_rec_hub（生产者）          waimai_rank_msg_processor（消费者）
负责：                              负责：
· 处理真实用户请求                  · 订阅 MQ topic
· 决策是否重放                      · 路由到对应消费者
· 序列化请求发到 MQ                 · 注入重放标记
· 异步调回 rec-hub

三、关键点

1. MessageWrapper 信封设计

class MessageWrapper {
String className;   // 请求类型，用于路由
String requestStr;  // Thrift 序列化的请求体
}

一个 topic 承载多条业务线，不需要为每个业务单独建 topic。className 是路由的唯一依据，新增业务线只需加 Consumer + Factory 一行路由，topic 不变。

---
2. 两层序列化

发送侧：
XxxRequest（Java对象）
↓ LogCompressToolV2.thriftCompress()  → Thrift二进制字符串
↓ JsonUtils.toStr(MessageWrapper)     → JSON字符串
写入 MQ

接收侧：
JSON字符串
↓ JsonUtils.toObj()                   → MessageWrapper
↓ LogCompressToolV2.thriftUncompress() → XxxRequest（原地填充）
恢复原始请求

Thrift 序列化保留字段类型和结构，比 JSON 更紧凑，适合结构复杂的排序请求。

---
3. 采样机制

MCC modelResetRatio = {"sg_indoor_rec_tab": 3000}
↓ 热更新（无需重启）
hashVal = murmur3(seed=7, bizTraceId) % 10000 + 1  → [1, 10000]
hashVal <= ratio → 命中（采样率 = ratio/10000 = 30%）

- 确定性：同一 bizTraceId 永远得到同一 hashVal，重试不会重复发送
- 均匀性：Murmur3 保证 hashVal 均匀散落，采样无偏
- 热更新：MCC 推送后 volatile 字段立即可见，调整采样率无需重启

---
4. 防死循环：resetFlag

rec-hub 收到重放请求
↓ context.isResetFlag() = true
↓ enableResetReq() return false
不再发送到 MQ

重放请求带着 resetFlag=1，rec-hub 识别后跳过采样判断，打破循环。

---
5. 双层异步解耦

主流程（用户请求）：同步，毫秒级返回
↓ 不等待
ResetReqProducerExecutor（8~10线程）：异步发 MQ
↓ 不等待
RankJobThreadPool（5~20线程）：异步调 rec-hub

用户感知零延迟，两个异步层各自隔离，互不影响。

---
6. 故障不扩散

// ModelResetConsumer：所有异常返回成功，MQ 不重试
catch (Exception e) {
status = ConsumeStatus.CONSUME_SUCCESS;
}

// XxxConsumer：消费失败直接丢弃，不影响主流程
return ConsumeStatus.CONSUME_SUCCESS;

重放是旁路流量，失败不需要补偿，永远返回 CONSUME_SUCCESS 防止 MQ 无效重试。

---
四、难点

难点1：采样的确定性与均匀性平衡

用随机数采样简单，但同一请求重试会重复发送。用 bizTraceId + Murmur3 既保证确定性（同一请求结果稳定），又保证均匀性（整体采样率准确）。两者兼顾是设计难点。

难点2：防止重放请求被二次重放

重放请求和真实请求走同一套排序流程，如果不加识别，rec-hub 处理完重放请求后会再次触发 enableResetReq，造成无限循环。通过 resetFlag=1 注入 + isResetFlag() 前置拦截打破循环，需要在消费端注入、在生产端识别两处配合。

难点3：多业务线共用一个 topic 的路由

不同业务线（PHF/BList/MED/SG）的请求结构完全不同，但共用一个 topic。MessageWrapper 的 className 字段是关键，消费端必须在反序列化前就知道用哪个类来解析，否则 Thrift 无法还原正确的对象结构。

难点4：连接池隔离

多个业务线共用 hubClientConnectionPoolConfig，但每个 bean（recHubSkuService、recHubMedService 等）各自持有独立的连接池实例。一条业务线流量暴涨不会抢占其他业务线的连接，这依赖于 Spring bean 的实例化机制，需要正确理解配置与实例的关系。
