采样完整逻辑

一、MCC 配置

MCC 中存两个配置项，都是 JSON 字符串：

// modelResetRatio：每个 scene 的采样阈值
{
"sg_indoor_rec_tab": 5000,
"med_tab": 3000
}

// modelResetTime：每个 scene 允许重放的时间窗口
{
"sg_indoor_rec_tab": [{"startTime": "08:00", "endTime": "22:00"}],
"med_tab": [{"startTime": "00:00", "endTime": "23:59"}]
}

服务启动后，MCC 每次推送变更，ModelResetRatioConvert 和 ModelResetTimeConvert 自动将 JSON 解析成 Map 注入到内存，无需重启立即生效：

@MtConfig(key = "modelResetRatio", converter = ModelResetRatioConvert.class)
private volatile static Map<String, Integer> MODEL_RESET_RATIO = new HashMap<>();

@MtConfig(key = "modelResetTime", converter = ModelResetTimeConvert.class)
private volatile static Map<String, List<TimeScope>> MODEL_RESET_TIME = new HashMap<>();

二、每次请求的完整判断流程

public static boolean enableResetReq(BaseContext context) {

    // 第一关：前置过滤，直接拦截不参与采样的请求
    if (context.isResetFlag()) return false;   // 本身是重放请求，防止无限循环
    if (Tracer.isTest())       return false;   // 压测流量，不污染重放链路
    if (context.getOffset()!=0) return false;  // 翻页请求，只对第一页采样

    // 第二关：时间窗口过滤
    List<TimeScope> timeScope = ModelResetConstants.getModelResetTimeByScene(context.getScene());
    if (!isInTimeScope(timeScope)) return false;  // 当前时间不在配置窗口内

    // 第三关：采样判断
    int ratio   = ModelResetConstants.getModelResetRatioByScene(context.getScene());
    int hashVal = HashUtil.hashByStr(7, context.getBizTraceId());

    if (ratio == 0 || hashVal > ratio) return false;

    return true;
}

三、采样判断的核心原理

第一步：把 bizTraceId 转换成 [1, 10000] 之间的整数

// HashUtil.java
public static int hashByStr(int seed, String value) {
HashCode hashCode = Hashing.murmur3_128(seed)
.hashString(value, StandardCharsets.UTF_8);
return Math.abs(hashCode.asInt()) % 10000 + 1;
}

bizTraceId（任意字符串）
↓ Murmur3(seed=7)        → 均匀分布的大整数（约 21 亿种值）
↓ Math.abs() % 10000     → 映射到 [0, 9999]
↓ + 1                    → 平移到 [1, 10000]

结果：hashVal ∈ [1, 10000]

Murmur3 的均匀性保证：大量不同的 bizTraceId 经过计算后，[1, 10000] 的每个值被覆盖的频率大致相等，不会集中在某个区间。哈希碰撞（不同 bizTraceId 映射到同一个 hashVal）客观存在，但不影响整体均匀性。

第二步：用 hashVal 与 ratio 比较

[  1  ....  ratio  |  ratio+1  ....  10000  ]
←─── 命中区间 ───→  ←───── 未命中区间 ──────→
发送重放                 不发送

hashVal <= ratio  → 命中，发送到 MQ
hashVal >  ratio  → 未命中，不发送

第三步：为什么 ratio 能代表比例

因为 hashVal 均匀分布在 [1, 10000]，命中区间 [1, ratio] 共有 ratio 个值，占总区间 10000 个值的比例就是：

采样率 = ratio / 10000

┌───────┬────────────┬────────────────┐
│ ratio │  命中区间  │     采样率     │
├───────┼────────────┼────────────────┤
│ 0     │ 空         │ 0%，关闭重放   │
├───────┼────────────┼────────────────┤
│ 1000  │ [1, 1000]  │ 10%            │
├───────┼────────────┼────────────────┤
│ 5000  │ [1, 5000]  │ 50%            │
├───────┼────────────┼────────────────┤
│ 10000 │ [1, 10000] │ 100%，全量重放 │
└───────┴────────────┴────────────────┘

四、bizTraceId 的作用

bizTraceId 是每次用户请求的唯一标识，在整个调用链中保持不变。用它做 Hash Key 有两个目的：

① 确定性：同一个请求永远得到同一个采样结果

同一次请求重试：
第1次 bizTraceId="abc-123" → hashVal=3721 → 命中
第2次 bizTraceId="abc-123" → hashVal=3721 → 命中，但 isResetFlag=true 被前置过滤拦截

不会因为重试导致重复发送到 MQ。

② 均匀性：大量不同请求的 hashVal 均匀散落

请求A bizTraceId="aaa" → hashVal=1200
请求B bizTraceId="bbb" → hashVal=7800
请求C bizTraceId="ccc" → hashVal=3500
请求D bizTraceId="ddd" → hashVal=9100
...（均匀散落在 [1, 10000]）
