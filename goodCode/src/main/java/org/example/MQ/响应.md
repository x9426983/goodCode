ModelResetConsumer 消费的是 rec-hub 服务在处理真实用户请求时"镜像复制"出来的请求。

---
完整链路

真实用户请求
↓
rec-hub 排序服务（waimai_rec_hub）
│  正常处理，返回结果给用户
│
└─ ResetReqProducer.sendResetReq()  ← 旁路复制
│
│ 判断是否要发送（enableResetReq）：
│   ✗ 本身是重放请求（resetFlag=1）→ 不发
│   ✗ 压测流量（Tracer.isTest）→ 不发
│   ✗ 翻页请求（offset != 0）→ 不发
│   ✗ 不在配置的时间窗口内 → 不发
│   ✗ 按 bizTraceId Hash 采样不中 → 不发
│   ✓ 以上都通过 → 发送
│
│ 构造 MessageWrapper：
│   LogCompressToolV2.thriftCompress(originalRequest)
│   → className = "...MedRequest"
│   → requestStr = Thrift序列化内容
│   → JSON序列化整体
│
↓ 异步线程池发送（不阻塞主流程）
Mafka MQ
topic: topic_recom_model_link_info
↓
ModelResetConsumer（waimai_rank_msg_processor）
│
│ 注入 resetFlag=1
↓
rec-hub 排序服务（用新模型重新排序）
│ 结果丢弃（不返回给用户）

---
核心设计：影子流量（Shadow Traffic）

┌────────────┬──────────────┬────────────────────────┐
│            │    主流程    │        重放流程        │
├────────────┼──────────────┼────────────────────────┤
│ 触发方式   │ 真实用户请求 │ 主流程完成后旁路复制   │
├────────────┼──────────────┼────────────────────────┤
│ 结果去向   │ 返回给用户   │ 丢弃（只用于模型验证） │
├────────────┼──────────────┼────────────────────────┤
│ 对用户影响 │ 有           │ 无                     │
├────────────┼──────────────┼────────────────────────┤
│ resetFlag  │ 无           │ = "1"（防止二次复制）  │
└────────────┴──────────────┴────────────────────────┘

---
几个关键控制点

采样控制（不是所有请求都重放）：
// 按 bizTraceId 做 Hash，控制重放比例
int hashVal = HashUtil.hashByStr(ModelResetConstants.hashSeed, context.getBizTraceId());
if (hashVal > ModelResetConstants.getModelResetRatioByScene(context.getScene())) {
return false;  // 不在采样范围内，不发
}

时间窗口控制（只在特定时段重放）：
// 支持跨天时间段，如 20:00 - 次日 10:00
List<TimeScope> timeScope = ModelResetConstants.getModelResetTimeByScene(scene);
if (!isInTimeScope(timeScope)) return false;

BList 额外过滤（只重放推荐 tab 的主 feed 请求）：
// 过滤掉：筛选tab、橱窗点击、多人餐点击等非主feed请求
if (tabCode != null) return false;
if (isClickMixWindow || isClickMultipleMeal) return false;

---
一句话总结：rec-hub 在处理真实用户排序请求时，按采样率、时间窗口等条件，将请求复制一份发到 MQ；waimai_rank_msg_processor 消费这些复制请求，打上重放标记后再次调用 rec-hub，用于在生产环境中用真实流量验证新排序模型的效果，且完全不影响用户体验。
