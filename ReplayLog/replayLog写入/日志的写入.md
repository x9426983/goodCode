
## 一次请求的数据是否全部写入同一个 RiverReplayLog 实例？

是的，整个请求生命周期共享同一个实例。 原因在于生命周期与 ContextBase 完全绑定：

```
@Setter
@Getter
@Slf4j
public class ContextBase implements ExecutionEnvironment {

    @Getter
    private RiverReplayLog replayLog;

    public ContextBase() {// ContextBase.java:（构造函数）
    replayLog = new RiverReplayLog();   // 随 ContextBase 一起诞生
    ...
    }
}
```

```
@Service("riverService")
public class RiverServiceImpl implements RiverService.Iface {
    // replaylog. 1M日志会被日志收集器丢掉
    private static final int MAX_LOG_BYTE_LENGTH = 1024 * 1024;
    private static final Logger replayLogger = LoggerFactory.getLogger("logger_wmrec_river_replaylog");

    // replaylog 黑名单场景集合
    @MtConfig(clientId = "riverServer", key = "replay_log_scene_blacklist", converter = SetStringConfigConverter.class)
    private volatile static Set<String> replayLogSceneBlackListSet = Collections.emptySet();

    // relaylog 重放场景白名单
    @MtConfig(clientId = "riverServer", key = "replay_log_reset_scene_list", converter = SetStringConfigConverter.class)
    private volatile static Set<String> replayLogResetSceneListSet = Collections.emptySet();

    @MtConfig(clientId = "riverServer", key = "river_use_mars_index_list", converter = SetStringConfigConverter.class)
    public volatile static Set<String> riverUseMarsIndexSet = Collections.emptySet();
    
    private ContextBase initContextBase(Request request) {
        ContextBase contextBase = new ContextBase();// 每次请求 new 一个 ContextBase
        contextBase.setRequest(request);
        return contextBase;
    }
}
```

ContextBase 实现了 ExecutionEnvironment 接口，被作为执行环境传入图的所有算子。算子通过 contextBase.getReplayLog() 拿到的永远是同一个实例。Map 字段用 ConcurrentHashMap 保证并发安全：

private Map<String, Object> extendData = new ConcurrentHashMap<>();
private Map<String, List<Long>> originRecalls = new ConcurrentHashMap<>();

这样设计的原因是聚合：一次推荐请求跨越几十个算子，日志需要汇总成一条完整的链路记录，而不是每个算子各打各的。
